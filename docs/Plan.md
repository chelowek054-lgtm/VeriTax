Ниже представлен **MVP-план** реализации проекта, сформулированный исключительно в архитектурных терминах. План описывает минимально необходимый набор функциональных блоков, достаточный для запуска системы и получения первого полезного результата, с учетом возможности последующего масштабирования без переписывания ядра.

---

### Общая структура MVP (5 обязательных модулей)

Для запуска требуются все пять описанных ранее архитектурных сущностей, но их внутреннее наполнение урезано до минимально жизнеспособного уровня.

1. **Модуль 1 (Ingestion MVP)**
2. **Модуль 2 (Enrichment MVP)**
3. **Модуль 3 (Storage MVP)**
4. **Модуль 4 (Admin MVP)**
5. **Модуль 5 (API MVP)**

---

### Модуль 1: Сбор данных (Ingestion Layer) — MVP

**Цель:** Стабильное получение сообщений из **одного** источника и их сохранение в неизменном виде.

| Пункт MVP | Описание наполнения | Задел на масштабирование |
| :--- | :--- | :--- |
| **1.1. Единый адаптер** | Поддержка **только Telegram** (как основной источник обсуждений по условию задачи). Адаптер умеет читать текст, подписи к медиа и цепочки ответов (`reply_to_message_id`). | Интерфейс адаптера должен быть абстрактным, чтобы позже можно было подключить Discord или RSS без изменения ядра. |
| **1.2. Сырое хранилище** | Запись каждого входящего события в **неизменяемый лог** в формате JSON (поля: `id`, `text`, `date`, `reply_to`, `media_caption`). | Хранилище — это просто файловое хранилище или одна таблица. Главное — накопление истории для возможного пересчета логики позже. |
| **1.3. Восстановление иерархии** | Проставление признака `thread_root_id` для каждого сообщения. Если `reply_to` указывает на сообщение, которого нет в логе, присваивается флаг `broken_context`. | Связывание сообщений в цепочки необходимо для будущего анализа контекста в Модуле 2. |
| **1.4. Медиа-прокси** | При наличии изображения/файла — сохранение **только ссылки** на файл в API Telegram (без скачивания файла на свои серверы). | В MVP не тратим ресурсы на хранение тяжелых файлов. Ссылки на оригиналы хранятся в логе. |

---

### Модуль 2: Обогащение и фактологизация (Enrichment Layer) — MVP

**Цель:** Превратить релевантное сообщение в простейший структурированный факт с каноническим именем модели.

| Пункт MVP | Описание наполнения | Задел на масштабирование |
| :--- | :--- | :--- |
| **2.1. Фильтр релевантности** | **Статический список ключевых слов** (Flux, Stable Diffusion, Midjourney, Kandinsky, SD3, генерация, img2img и т.д.). Если ни одного слова нет — сообщение игнорируется. | В будущем замена на легковесный классификатор LLM. Ключевые слова в MVP дают быстрый отсев 70% мусора. |
| **2.2. Извлечение сущностей (Regex-based)** | Поиск в тексте **заранее заданного списка моделей** (например, `Flux|SD3|Midjourney`). Если найдено несколько — создается факт для каждой. | В будущем замена на NER (Named Entity Recognition) через LLM. |
| **2.3. Канонизация имен** | **Жесткий словарь синонимов** (`"флюкс" -> "Flux"`, `"миджорни" -> "Midjourney"`). Неизвестные названия **игнорируются** (не попадают в базу). | В будущем замена на Карантин и ручное связывание в Админке. В MVP пропускаем только то, что знаем. |
| **2.4. Анализ тональности** | **Примитивная эвристика**: наличие смайлов 👍/🔥 -> позитив, 👎/💩 -> негатив. Отсутствие -> нейтрально. | В будущем замена на sentiment-модель. Эвристики достаточно для первых статданных. |
| **2.5. Формирование Факта** | Объект: `{ model: "Flux", sentiment: "pos", source_id: "msg_123", thread_id: "thread_456", broken: false }`. | Структура Факта должна быть расширяемой (поле `attributes`). В MVP заполняется только `sentiment`. |

---

### Модуль 3: Хранилище знаний (Storage Layer) — MVP

**Цель:** Сохранить факты так, чтобы можно было быстро посчитать статистику по конкретной модели.

| Пункт MVP | Описание наполнения | Задел на масштабирование |
| :--- | :--- | :--- |
| **3.1. Хранение Фактов (Табличное)** | **Реляционная таблица `facts`** с колонками: `model_name`, `sentiment`, `source_msg_id`, `thread_id`, `created_at`. | В будущем миграция на Графовую БД. Сейчас достаточно SQL, чтобы делать `GROUP BY model_name`. |
| **3.2. Агрегация "на лету"** | Нет готовых агрегатов. При запросе статистики выполняется SQL-запрос: `SELECT model_name, COUNT(*) as mentions, SUM(CASE sentiment WHEN 'pos' THEN 1 ELSE 0 END) as pos FROM facts WHERE created_at > now() - interval '7 days' GROUP BY model_name`. | В будущем добавление материализованных представлений. |
| **3.3. Учет веса сообщений** | Вводится поле `weight` в таблице `facts`. По умолчанию = 1.0. Если `broken_context = true` -> `weight = 0.3`. | Логика расчета веса инкапсулирована в Модуль 2. Модуль 3 просто хранит число. |
| **3.4. Векторный поиск** | **Отсутствует в MVP**. | В будущем добавится отдельный компонент для эмбеддингов веток. В MVP пользователь ищет только по названию модели. |

---

### Модуль 4: Управление (Admin Panel) — MVP

**Цель:** Дать возможность вручную править словарь синонимов и смотреть сырые данные для отладки.

| Пункт MVP | Описание наполнения | Задел на масштабирование |
| :--- | :--- | :--- |
| **4.1. Редактор синонимов** | Простая форма: "Сырое значение" -> "Каноническая модель". Админ может добавить правило `"SD 3.5" -> "Stable Diffusion 3.5"`. | В будущем превратится в управление Карантином и Таксономией. |
| **4.2. Просмотр сырых фактов** | Таблица с пагинацией: последние N записей из Модуля 3. Фильтр по модели и тональности. | В будущем — полноценная лента модерации и интерфейс разрешения конфликтов. |
| **4.3. Ручной отзыв факта** | Кнопка "Пометить как спам/ошибку". Меняет флаг `is_valid` в таблице `facts` на `false` (факт исключается из агрегации). | Механизм версионности и опровержений заложен через поле `is_valid`. |

---

### Модуль 5: API и Витрина (API MVP)

**Цель:** Выдать первую полезную статистику в ответ на прямой вопрос.

| Пункт MVP | Описание наполнения | Задел на масштабирование |
| :--- | :--- | :--- |
| **5.1. Единственный эндпоинт** | `GET /api/v1/stats?period=week` | В будущем добавятся фильтры `?model=...&hardware=...`. |
| **5.2. Формат ответа** | JSON со списком моделей и счетчиками: `{ "models": [ { "name": "Flux", "pos": 45, "neg": 2, "total": 47 } ] }`. | Структура ответа должна предусматривать поле `sources` для ссылок. |
| **5.3. Ссылки на источники** | Поле `source_msg_id` в ответе **не отдается** (в целях MVP). Отдается только агрегат. | В будущем — формирование прямой ссылки на сообщение в Телеграме. |
| **5.4. Генерация текста (LLM)** | **Отсутствует**. Ответ — сухие цифры в JSON. | В будущем оборачивается LLM-шапкой с человеческим языком. |

### Итоговая схема взаимодействия в MVP

```text
[Telegram Chat] 
      │ (Новое сообщение)
      ▼
[Модуль 1] -> Сохранили в лог -> Определили `thread_root` и `broken` -> Отправили дальше.
      │
      ▼
[Модуль 2] -> Есть ключевое слово? -> Извлекли модель по словарю -> Оценили смайлик -> Создали объект Факта.
      │
      ▼
[Модуль 3] -> Записали строку в таблицу `facts` с полями (model, sentiment, weight).
      │
      ▼
[Модуль 4] -> Админ зашел в панель, добавил синоним "кАндинский" -> "Kandinsky".
      │
      ▼
[Модуль 5] -> Пользователь открыл /stats -> SQL GROUP BY -> Показал JSON.
```

Этот план обеспечивает получение **первого измеримого результата** (статистика популярности моделей) с минимальными архитектурными затратами и закладывает правильную структуру данных для последующего усложнения до полноценного RAG с графами и таксономией.
