# Список 1: Модульное ядро системы (Архитектурные обязанности)

## Модуль 1. Слой сбора и нормализации данных (Ingestion Layer)

**Цель:** Привести любые внешние источники (Telegram, Discord, Форумы, RSS) к единому внутреннему формату «Событие» и обеспечить сохранность исходного контекста.

### Обязанности

- Подключение к источникам: Реализация адаптеров для подключения к разнородным API (мессенджеры, веб-хуки).
- Нормализация формата: Преобразование всех входящих данных в объект `RawEvent` с едиными полями: `source_id`, `timestamp`, `author_id`, `content_type` (`text`, `image`, `file`), `reply_structure`.
- Восстановление иерархии: Формирование цепочек «родитель-потомок» на основе метаданных платформы (`reply-to`, цитирование) даже при неполных данных.
- Сырое долговременное хранение: Сохранение точной копии `RawEvent` в неизменяемое архивное хранилище (`Data Lake`) для возможности переиндексации в будущем.
- Грубая первичная фильтрация: Отсечение заведомо пустых событий (пустой текст без медиа), но с сохранением их в архиве с пометкой `ignored`.
- Скачивание медиа: Перенос изображений и файлов из временных ссылок источника в контролируемое объектное хранилище системы.

## Модуль 2. Слой обогащения и фактологизации (The "Euler" & Taxonomy Layer)

**Цель:** Превратить неструктурированный поток нормализованных событий в семантически точные, пригодные для машинной обработки «Атомарные Факты» и связать их с таксономическим деревом.

### Обязанности

- Фильтрация по релевантности: Оценка принадлежности события к предметной области (нейросети, генерация изображений) с помощью классификатора намерений. Отсев оффтопа до этапа дорогой обработки.
- Разрешение имен (Name Resolution): Приведение сленговых названий и опечаток («флюкс», «миджорни») к единому каноническому идентификатору узла таксономии. Направление неизвестных сущностей в карантин.

### Извлечение структурированной информации

- Выделение именованных сущностей (модели, железо, софт).
- Выделение числовых атрибутов с их экранированием от искажений (цена, объем памяти, скорость).
- Определение тональной окраски относительно каждой сущности.

- Генерация Атомарного Факта: Создание объекта `Fact`, включающего: `канонический_узел`, `атрибут`, `значение`, `вес_достоверности`.
- Обработка «битого контекста»: Ослабление веса достоверности для событий, входящих в цепочки с удаленными или отсутствующими родительскими сообщениями.
- Таксономическое картирование: Привязка сущности к позиции в иерархическом дереве знаний («Царства и Классы»), обновление структуры дерева при появлении новых типов объектов (задача для модератора).

## Модуль 3. Слой хранения знаний (Storage Layer: Граф + Вектор)

**Цель:** Обеспечить высокоуровневую навигацию по знаниям (кто с кем связан) и семантический поиск похожих обсуждений без потери данных из-за дублирования или пересечений.

### Обязанности

### Графовое представление (Структура)

- Хранение узлов-сущностей (Модели, Пользователи, Ветки обсуждений).
- Хранение ребер-свидетельств, соединяющих Пользователей с Узлами-сущностями через Факты.
- Управление весом ребер в зависимости от авторитетности источника, свежести и подтверждений.

### Векторное представление (Семантика)

- Создание и хранение эмбеддингов только для агрегированных резюме обсуждений, а не для каждого сообщения.
- Индексация для поиска вопросов, схожих по смыслу с запросом пользователя.

### Реализация логики пересечений (Круги Эйлера)

- Управление связями «многие-ко-многим» через мета-узлы контекста (например, узел «Ветка обсуждения X» связывает сразу несколько фактов).
- Избегание создания дубликатов узлов-фактов при поступлении новых свидетельств о том же самом.

### Управление версионностью и опровержениями

- Сохранение истории изменений мнений (ребро `WAS_VALID -> IS_RETRACTED`).
- Возможность пометить узел или ребро как устаревшее без физического удаления данных.
- Обеспечение ссылочной целостности: Гарантия, что каждый `Fact` ссылается на исходный `RawEvent` из Модуля 1.

## Модуль 4. Слой управления и факт-чекинга (Governance & Admin Panel)

**Цель:** Предоставить интерфейс для вмешательства человека в автоматические процессы с целью исправления ошибок таксономии, верификации спорных фактов и контроля качества базы.

### Обязанности

- Интерфейс работы с Карантином: Отображение списка нераспознанных сущностей и предложений от Модуля 2 для ручного связывания или создания новых узлов.
- Интерфейс разрешения конфликтов: Отображение пар противоречащих друг другу Фактов с предложением выбрать истинный или зафиксировать состояние «Противоречие в источниках».
- Редактор таксономии: Визуальное управление иерархическим деревом (Drag-and-Drop узлов модели из одной категории в другую) с автоматическим пересчетом связей в графе.
- Аудирование операций: Логирование всех действий администратора (кто, когда и почему переопределил факт).
- Настройка правил фильтрации: Управление списком стоп-слов, доменов релевантности и весовыми коэффициентами для источников.

## Модуль 5. Слой доступа (API и Публичная витрина)

**Цель:** Предоставить конечному пользователю инструменты для получения агрегированной, достоверной статистики и глубокого погружения в досье объектов.

### Обязанности

- Обработка нечетких запросов: Парсинг вопроса пользователя («Что лучше для 3090?»), извлечение фильтров и намерений.

### Агрегация данных по запросу

- Выборка из Модуля 3 взвешенных фактов по заданным узлам.
- Формирование рейтингов и списков «Топ N».

### Генерация аргументированного ответа

- Формирование текстового резюме на основе агрегированных данных.
- Обязательное включение ссылок на исходные свидетельства (`RawEvent`) для верификации.

### Генерация страницы «Вики / Досье»

- Сборка всех известных системе атрибутов, статистики упоминаний и связанных обсуждений об объекте (модели/технологии).
- Отображение хронологии изменения мнений и маркеров достоверности.
- Интерактивное уточнение: Предоставление интерфейса (кнопки/фильтры) для сужения контекста запроса (например, переход от «Flux» к «Flux на RTX 3060»).

# Список 2: Критически важные тонкости и ключевые механики (по модулям)

## Модуль 1: Важное

- Архивность превыше всего: Нельзя удалять сырые данные даже после их обработки. При изменении логики Модуля 2 потребуется ретроспективный пересчет.
- Борьба с удалением: Система должна быть готова к тому, что `reply_to` может указывать на несуществующий ID. В этом случае ветка не игнорируется, но достоверность её фактов искусственно занижается.
- Адаптеры не должны быть умными: Модуль 1 не должен пытаться анализировать текст. Его задача - точная и быстрая перекладка байтов.

## Модуль 2: Важное

- Экранирование чисел (Rigid Extraction): LLM склонна округлять значения. В промпт или пайплайн должен быть вшит детерминированный парсер, вырезающий цифры до подачи в нейросеть и вставляющий их обратно после суммаризации как `<NUM_LOCKED>`.
- Карантин неопределенных имен: Нельзя привязывать факт к «похожему» имени автоматически. Лучше недодать информации, чем перепутать Flux Pro и Flux Dev. Связывание только через подтверждение в Модуле 4.
- Вес битого контекста: Факт, извлеченный из диалога, где было удалено сообщение собеседника, должен иметь понижающий коэффициент `0.3` при расчете агрегаций в Модуле 5.

## Модуль 3: Важное

- Разделение «Факта» и «Упоминания»: Узел-факт («Flux быстрый») один. Ребер-свидетельств к нему - тысячи. Это единственный способ удержать граф от взрывного роста и считать статистику.
- Реализация Эйлерова пересечения: При запросе «Flux + Качество» мы не ищем отдельный узел «Flux-Качество», а ищем пересечение множества `INCOMING_LINKS(Flux)` и `INCOMING_LINKS(Качество)`. Мета-узел «Обсуждение» является той областью на диаграмме Венна, где эти множества пересекаются.
- Отзыв факта ≠ Удаление: При обнаружении ошибки в графе ставится флаг `RETRACTED`. Это важно для исторической справки: «Почему 10 апреля все считали модель А плохой? - Потому что вышла статья с ошибкой, позже опровергнутая».

## Модуль 4: Важное

- Защита от административного хаоса: Редактирование таксономии (перенос узла в графе) - транзакционная операция. Она должна автоматически пересчитать все дочерние связи, чтобы не порвать базу знаний.
- Разрешение конфликтов: Админ должен иметь возможность не только выбрать «Правду А или Б», но и зафиксировать мета-факт: «Мнения разделились: 60% за А, 40% за Б». Это ценнее, чем фальшивый консенсус.

## Модуль 5: Важное

- Ответ всегда с доказательствами: Пользователю нельзя выдавать просто текст «Лучшая модель - Flux». Ответ обязан содержать строку: «Сформировано на основе анализа X сообщений за период Y. Показатель согласия: Z%».
- Управление ожиданиями: Если для ответа на запрос «Для RTX 3050» недостаточно фактов, API должен честно ответить: «По вашему запросу найдено мало обсуждений (всего 3 сообщения). Точность прогноза низкая. Рекомендуем ознакомиться с общими обсуждениями: [ссылка]».
- Кнопка «Узнать всё» = Снапшот графа: Генерация вики-страницы - это не вызов LLM, а шаблонизированная выгрузка структуры связей из Модуля 3 на момент запроса. Это гарантирует, что досье соответствует актуальному состоянию базы.
