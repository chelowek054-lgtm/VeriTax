# Фаза 0: Источники данных (Внешний контур)

```text
[Telegram] [Discord] [RSS] [API Webhook] ...
         │
         ▼
   [Модуль 1: Ingestion Layer]
```

## Фаза 1: Сбор и нормализация (Модуль 1)

```text
1. Адаптер источника принимает сырое сообщение (текст/медиа/метаданные).
        │
        ▼
2. Нормализация в объект `RawEvent`:
   - Приведение timestamp к UTC.
   - Извлечение цепочки `reply_to` (восстановление иерархии).
   - Сохранение исходного контента без изменений.
        │
        ├─────────────────────────────┐
        │                             │
        ▼                             ▼
3a. Сырое архивное хранилище    3b. Временное объектное хранилище
    (Data Lake)                     (для медиафайлов)
    - Только запись.                - Скачивание картинок/файлов.
    - Ключ: `{source}_{message_id}` - Генерация внутреннего URL.
        │                             │
        └─────────────┬───────────────┘
                      ▼
4. Первичная фильтрация (Rule-based):
   - Если (длина текста < N) И (нет медиа) -> Пометить `IGNORED`.
   - Если (текст содержит стоп-слова) -> Пометить `SPAM`.
   - Иначе -> Статус `ACCEPTED`.
        │
        ▼
5. Публикация события `NormalizedEvent` в шину данных для Модуля 2.
```

## Фаза 2: Обогащение и фактологизация (Модуль 2)

```text
1. Получение `NormalizedEvent` из шины.
        │
        ▼
2. Классификатор релевантности (Легкая LLM / Правила):
   - Вопрос: "Относится ли сообщение к ИИ/нейросетям/генерации изображений?"
        │
        ├─── НЕТ (confidence < порог) ───► [Конец обработки]
        │    (Событие архивируется с тегом `IRRELEVANT`)
        │
        └─── ДА ───► Продолжение обработки.
                      │
                      ▼
3. Предобработка текста перед LLM-анализом:
   - Детерминированный парсер чисел:
     Поиск паттернов `$XXX`, `XX GB`, `XX sec`.
     Замена на теги `<NUM_LOCKED id=1>24</NUM_LOCKED>`.
        │
        ▼
4. Извлечение сущностей и атрибутов (Основная LLM):
   - Выделение именованных сущностей (Flux, SD3.5, 3090).
   - Определение атрибута (качество, скорость, цена).
   - Определение тональности (positive/negative/neutral).
        │
        ▼
5. Шлюз разрешения имен (Name Resolution):
   - Поиск в "Реестре канонических имен".
        │
        ├─── НАЙДЕНО ───► Замена на канонический ID (например, `model:flux_pro_1.1`).
        │
        └─── НЕ НАЙДЕНО ───► Создание "Сущности-кандидата" и отправка в КАРАНТИН (Модуль 4).
                              Дальнейшая обработка этого атрибута приостанавливается до верификации.
                              │
                              ▼
                         [Модуль 4: Карантин]
                              │
                              └───► (После ручного связывания) ───► Возврат в поток (шаг 5).
        │
        ▼
6. Формирование объекта `AtomicFact`:
   {
     canonical_entity_id: "model:flux_pro",
     attribute: "vram_requirement",
     value: "24 GB",
     sentiment: "neutral",
     confidence: 0.85,
     source_raw_event_id: "...",
     context_broken_flag: true/false
   }
        │
        ▼
7. Публикация `AtomicFact` в шину для Модуля 3.
```

## Фаза 3: Сохранение в граф и векторную БД (Модуль 3)

```text
1. Получение `AtomicFact` из шины.
        │
        ▼
2. Проверка на существование УЗЛА-ФАКТА в графе:
   - Ключ: `canonical_entity_id` + `attribute` + `value`.
        │
        ├─── УЗЕЛ НЕ НАЙДЕН ───► Создание нового узла `FactNode`.
        │                        (Например, "Flux Pro требует 24 ГБ VRAM")
        │
        └─── УЗЕЛ НАЙДЕН ───► Использование существующего узла.
                              │
                              ▼
3. Создание РЕБРА-СВИДЕТЕЛЬСТВА:
   - Тип ребра: `MENTIONS` или `CONFIRMS` (в зависимости от тональности).
   - Соединение: `(User Node) -[:CONFIRMS {date, weight, raw_ref}]-> (FactNode)`.
   - Расчет веса ребра:
     base_weight = 1.0
     if context_broken_flag: weight *= 0.3
     if user_is_expert: weight *= 1.5
        │
        ▼
4. Обработка противоречий (Conflict Resolution):
   - Поиск существующих ребер с ПРОТИВОПОЛОЖНОЙ тональностью от этого же факта.
        │
        ├─── ПРОТИВОРЕЧИЕ ОБНАРУЖЕНО ───►
        │    - Создание алерта для Модуля 4 ("Требуется разрешение конфликта").
        │    - Ребро все равно создается, но с пониженным весом и флагом `contested: true`.
        │
        └─── ПРОТИВОРЕЧИЙ НЕТ ───► Обычное сохранение.
                                   │
                                   ▼
5. Создание / Обновление МЕТА-УЗЛА КОНТЕКСТА (Обсуждения):
   - Если у `AtomicFact` есть ссылка на ветку (`thread_id`), создается/находится узел `Discussion`.
   - Устанавливаются связи: `(Discussion) -[:INCLUDES_FACT]-> (FactNode)`.
   - Это реализует логику Эйлерова пересечения.
        │
        ▼
6. Векторное индексирование (Асинхронно / По расписанию):
   - Только для **агрегированных контекстов**:
     Раз в N часов собираются все новые факты в рамках `Discussion`,
     генерируется краткое резюме обсуждения, и **оно** векторизуется.
   - Вектор сохраняется с привязкой к узлу `Discussion`.
        │
        ▼
7. [Конец цикла записи]. Данные готовы к запросам.
```

## Фаза 4: Обработка пользовательского запроса (Модуль 5 + Модуль 3)

```text
1. Пользователь вводит запрос: "Какая модель лучше для 3090?"
        │
        ▼
2. API (Модуль 5) -> Парсинг намерения:
   - Извлечение фильтров: { hardware: "RTX 3090", domain: "image_gen" }.
   - Тип запроса: COMPARISON.
        │
        ▼
3. Семантический поиск (Векторная БД Модуля 3):
   - Векторизация запроса.
   - Поиск похожих обсуждений (узлов `Discussion`) за последние 7-30 дней.
        │
        ▼
4. Запрос к Графу (Модуль 3):
   - Найти все модели (`ModelNode`), которые связаны с `HardwareNode: RTX_3090`.
   - Для каждой модели просуммировать **веса** входящих ребер-свидетельств `CONFIRMS`
     с положительной тональностью.
   - Ранжирование моделей по сумме весов.
        │
        ▼
5. Извлечение подтверждающих фактов (Evidence Retrieval):
   - Для топ-3 моделей извлечь `AtomicFact` с наибольшим весом.
   - Получить `raw_event_id` для формирования ссылок на источник.
        │
        ▼
6. Генерация ответа (LLM Модуля 5):
   - Промпт содержит **только** агрегированные данные из шага 4 и 5.
   - Инструкция: "Не выдумывай. Если данных мало, скажи об этом."
        │
        ▼
7. Формирование финального ответа пользователю:
   {
     "summary": "На основе 45 обсуждений за неделю, лидер для RTX 3090 - SD3.5 Medium (требует 8GB VRAM).",
     "confidence": "Высокая (N=45)",
     "models": [
       { "name": "SD3.5 Medium", "score": 92, "sources": ["msg_123", "msg_456"] },
       { "name": "Flux Schnell", "score": 78, "sources": ["msg_789"] }
     ],
     "disclaimer": "Данные по Flux Pro противоречивы. [Посмотреть спорные моменты]"
   }
        │
        ├──────────────────────────────┐
        │                              │
        ▼                              ▼
8. Отображение в UI              9. Кнопка "Узнать всё о SD3.5 Medium"
   - Ссылки кликабельны.             │
   - Кнопка уточнения.               ▼
                               10. Генерация Вики-страницы (Модуль 5 -> Модуль 3):
                                   - Сбор всех узлов, связанных с `SD3.5 Medium`
                                     (атрибуты, обсуждения, хронология).
                                   - Шаблонизация данных в статичную карточку.
                                   - Отображение графика "История упоминаний".
```

## Фаза 5: Управление и коррекция (Модуль 4 - Асинхронный контур)

```text
[Администратор входит в Панель управления]
        │
        ├───► Просмотр "Карантина":
        │     - Список нераспознанных сущностей (шаг 5 Фазы 2).
        │     - Действие: "Связать с существующей моделью" или "Создать новую".
        │     - После действия → Обновление Реестра канонических имен.
        │
        ├───► Просмотр "Конфликтов":
        │     - Список FactNode, где есть ребра с противоположной тональностью.
        │     - Действие: "Подтвердить версию А", "Подтвердить версию Б", "Зафиксировать как спор".
        │     - После действия → Простановка флага `RETRACTED` на неверное ребро и пересчет агрегаций.
        │
        └───► Редактор таксономии:
              - Drag-and-Drop узла "Flux" из папки "DiT" в папку "Рекомендованные 2026".
              - Транзакционное обновление свойства `category` у узла в графе.
              - Сброс кеша агрегаций (чтобы в следующих запросах "лучшая модель" учитывалась с новой категорией).
```
